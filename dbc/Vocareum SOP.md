# Vocareum SOP Appendices

**DEVOPS NOTE:** the behaviour documented here applies to the integration code maintained in the VOCAREUM PARENT SOURCE course container. Please clone this course to benefit from the behaviour outlined in this document.

## Course Folder Location

For the time being, course material (DBCs, configuration files, etc) can be stored in one of two places:

* **/voc/private/courseware** Put courseware here. If you clone the course or assignment, the courseware will follow and work without modification.  
* **\[DEPRECATED\] /voc/course/partXXXXXXX**. This approach continues to be supported but is no longer the preferred method since it introduces pain points when cloning courses. It requires a specific number matching up with the part id assigned by Vocareum. These folders are visible to all assignments in the course. 

## Course Folder Structure

### Metadata

Course metadata is specified using one (or more) JSON-formatted configuration files in the course folder. There are no specific naming requirements for the configuration file(s) but they must end with the suffix **.json** or **.cfg**. For example: *data-engineering-with-databricks.json*  
The course metadata specifies all the information needed to bootstrap the course. For example, the DBC containing the lab notebooks (and hence Workspace-setup notebook), cluster configuration, grading notebooks (if the course supports autograding), and more.  
For a description of the parameters that can be specified in the configuration, refer to the [Configuration FIle Options](#configuration-file-options) appendix.

### DBC/Zip Files

Most courses will deliver material in the form of DBC files; archives containing Databricks notebooks. Alternatively, source .zip files may be used (as generated by the build tool or when you export a folder from a workspace using the “source” option). DBC or zip files may be laid directly into the course folder. There are no specific naming requirements for these files, as you specify the name of this file in the configuration file.

### Patches

DBCs can be “hot-patched” if it’s not practical to rebuild it for whatever reason. Patching allows you to add or replacing one (or more) files or notebooks in a DBC. There are no strict naming/structural requirements surrounding patches, but the following conventions are suggested:

* Place the replacement notebooks in a subfolder named **patches**.  
* For easier maintenance, name the files in that subfolder after the files they replace.

### Data sideloading

Files can be injected into the environment using this alternative to Marketplace-driven datasets. This is a good option for small to medium sized datasets that aren’t suitable for publishing in Marketplace.

Any files found in the course directory meeting the following criteria will be sideloaded (that is, the file tree will be traversed, and any files within will be injected into the environment):

- Any archives conforming to the filename patter ***files-\*.zip***  
- Folder named ***files*** (if present); this is potentially more convenient if you only have a few files to sideload and wish to avoid building them into an archive

The path structure within the archive/folder dictates where in the environment the files will be landed. So, for example, 

`📄 files-example.zip` (or `files/)`  
`│`  
`📂 Volumes                    (Only supported top-level folder atm)`  
`└── 📁 {catalog_name}         (Catalog: any name or DEFAULT)`  
    `└── 📁 {schema_name}      (Schama: choose whatever name you want)`  
        `└── 📁 {volume_name}  (Volume: choose whatever name you want)`  
            `├── 📄 ex1.csv`  
            `├── 📄 ex2.csv`  
            `└── 📁 ex1.delta`  
                `└── 📄 …`

In this example:

* The file structure in {volume\_name} will be copied in place into a volume with the same name  
* That volume will be created in a schema named {schema\_name}  
* That schema will be created in the catalog named {catalog\_name} (note, if you specify *DEFAULT* here, it will be created in the default catalog, typically *dbacademy*)  
* All users will have read access to these resources

### Patches

DBCs can be “hot-patched” if it’s not practical to rebuild it for whatever reason. Patching allows you to add or replacing one (or more) files or notebooks in a DBC. There are no strict naming/structural requirements surrounding patches, but the following conventions are suggested:

* Place the replacement notebooks in a subfolder named **patches**.  
* For easier maintenance, name the files in that subfolder after the files they replace.

## Configuration File Options {#configuration-file-options}

Course metadata files are structured in JSON format. This section documents configuration parameters.

### DBC object

The DBC object represents a pattern used in the course configuration, and is used to describe a literal DBC file or a source .zip file. This object can have the following values.

* **src** (required): string  
  Specifies the source of the DBC/zip (can be an absolute file path or relative to the folder containing the course material), or a URL to a public git repository.  
* **provider** (optional): string  
  Specifies the git provider, if **src** is a git repository URL (defaults to GitHub if not specified).  
* **entry** (optional): string  
  Specifies the “main” notebook (the meaning of this is context-dependent). If not specified, the system will assume the first notebook in the alpha-sorted list.  
* **folder** (optional): string  
  Specifies the name for the top-level folder into which the material will be extracted. If not specified, defaults to the file name of the file itself.  
* **patch** (optional): list(object)  
  Replace or insert one or more notebooks or files after the import. Each object in the list takes the following:  
  * **target** (required): string  
    Destination path of notebook/file to insert or replace, relative to the top-level folder into which material was imported. If the target already exists, it is replaced.  
  * **src** (required): string  
    Path of replacement notebook/file. This path can be absolute or relative, similar to the specification of the DBC **src**.

### Metastore configuration

JSON key: **metastore\_config** (optional: object)  
This provides fine control over your course’s metastore allocation and configuration. Omitting this section from your configuration will trigger sensible default settings: a metastore will be allocated and shared across all workspaces associated with the course, and the metastore will contain a default catalog named “dbacademy” where all user schemas will reside by default. Users will not be able to create metastore-level constructs (like catalogs).

* **name** (optional and not recommended: str)  
  Provides a name for the metastore, and indirectly, dictates the level of isolation from other courses. For example, if specifying a unique name here, this metastore will only be shared between workspaces associated with the course. However, if using the same name across multiple courses, a “global shared” metastore concept can be implemented.  
  **If not specified**, defaults to a name incorporating the course id (that is, a per-course metastore is the default). **Typically, this is the desired behaviour.**  
* **default\_metastore\_name** (optional: str)  
  Specifies the name of the metastore from which to copy settings when instantiating new metastores. If not specified, will default to the first metastore it can find in the same cloud region.  
* **default\_catalog** (optional: str, defaults to “*dbacademy”*)  
  Provides a name for the default catalog. This is the catalog where:  
  * Operational data is maintained  
  * User schemas are created (under the default data architecture model)  
  * Configured as workspace-level default catalog (and default catalog for all classic clusters)  
* **unique** (optional: bool, defaults to false)  
  The default is that each workspace shares the same metastore. This option ensures each workspace has its own metastore. This option helps with Lakehouse monitoring which has a limit of 100 monitors per metastore.  
* **enable\_create\_catalog** (optional: bool, defaults to *False*)  
  Allow users to create catalogs. Unless the course has a specific requirement for this, we recommend leaving this turned off. If you turn it on, please do not specify a shared metastore (ie do not specify **name**).  
* **storage\_root** (optional: str)  
  Specifies the cloud location for storing managed tables and volumes in the metastore. If not specified, copies the value from the default metastore (see **default\_metastore\_name**).  
* **iam\_role** (optional: str)  
  Specifies the ARN of the IAM role used to authenticate with the cloud location specified by **storage\_root**. If not specified, copies the value from the default metastore (see **default\_metastore\_name**).  
* **owner** (optional: str)  
  Specifies the principal who should own the metastore and all critical assets. If not specified, copies the value from the default metastore (see **default\_metastore\_name**).  
* **system\_schemas** (optional: list(str))  
  Specifies system schemas to enable. Any enabled schemas will have USE SCHEMA and SELECT enabled for all users.


### Workspace configuration

JSON key: **workspace\_setup** (extends DBC object)

Specifies content to execute as part of the process of provisioning a new workspace for this lab. This setup is executed as a user with administrative privileges at the workspace and metastore level. A typical application of this is to run the “Workspace-setup” notebook found in many of our course DBCs. Note the following:

* Use **entry** to specify the notebook responsible for orchestrating workspace setup. This notebook will be run with the following options:  
  * **host**: specifies the workspace URL to use for issuing REST API calls, if your notebook needs to do that  
  * **token**: specifies a bearer token to use for REST API authentication, if your notebook needs to do that.  
* The additional **timeout\_minutes** parameter is supported in the setup block. By default, the system expects this notebook to complete within 20 minutes, but this parameter allows you to override that value.  
* Though the **folder** option is accepted and honored, it has no visible impact to the users of the system and its use here is not generally recommended.  
* Any supplied patches are applied prior to running this job.

### User configuration

JSON key: **user\_config**

Specifies user-specific configuration

* **catalog** (optional: str)  
  Specifies a catalog to create for the user. Though the user will not have ownership of the catalog, they will have ALL PRIVILEGES (unless it’s a shared catalog as discussed below).  
  If using this feature, the catalog name should be specified as one of the following values:  
  * **USERNAME**: catalog name is based on username to ensure uniqueness in the catalog namespace  
  * **RANDOM**: catalog name is based on random words to ensure uniqueness in the catalog namespace  
  * Any other strings will be interpreted as a literal. In this scenario, a shared catalog will be created that all users can access through a reduced set of privileges (though they can create schemas within the catalog).  
  * **username**: specifies the full user id (email) of the user being provisioned.  
* **manage\_catalog** (optional: boolean, defaults to false)  
  If a catalog is created for a user, this option additionally provides them with the MANAGE privilege, giving them ownership-like capabilities. For example, they can grant permissions on the catalog to others (useful for data governance use cases).  
* **schema** (optional: str)  
  Specifies name of schema to create for user. Though the user will not have ownership of the schema, they will have ALL PRIVILEGES (unless it’s a shared schema as discussed below). If using this feature, the schema name should be specified as one of the following values:  
  * **USERNAME**: schema name is based on username to ensure uniqueness in the namespace  
  * **RANDOM**: schema name is based on random words to ensure uniqueness in the namespace  
  * Any other strings will be interpreted as a literal. In this scenario, a common schema will be created that all users can access through a reduced set of privileges (though they can create objects within the schema).  
* **manage\_schema** (optional: boolean, defaults to false)  
  If a schema is created for a user, this option additionally provides them with the MANAGE privilege, giving them ownership-like capabilities. For example, they can grant permissions on the schema to others (useful for data governance use cases).  
* **cluster\_config** (optional: object)  
  If specified, the system will automatically create and manage a cluster for the user, applying the parameters specified in this block. Parameters accepted are outlined in the [Databricks documentation for cluster creation](https://docs.databricks.com/api/workspace/clusters/create). For parameters not supplied, the system will assume sensible defaults.  
  **Cluster naming**  
  Clusters will be named according to the setting of the **cluster\_name** parameter. Options for this parameter include the following values:  
  * **USERNAME**: cluster name is based on username to ensure uniqueness in the namespace  
  * **RANDOM**: cluster name is based on random words to ensure uniqueness in the namespace  
  * Any other string will be interpreted as a literal. In this scenario, a shared cluster will be created and accessible by all users.

  **Libraries**

  If you need libraries installed on the cluster, include a libraries object nested in the cluster\_config. Parameters of the libraries object are outlined in the [Databricks documentation for library addition](https://docs.databricks.com/api/workspace/libraries/install).

* **warehouse** (optional: object)  
  If specified, the system will automatically create and manage a SQL warehouse for each user, applying the parameters specified in this block. For a description of the parameters, refer to the [Databricks documentation for SQL warehouse creation](https://docs.databricks.com/api/workspace/warehouses/create). For parameters not supplied, the system will assume sensible defaults.  
  **Warehouse naming**  
  Warehouses will be named according to the setting of the **name** parameter. Options for this parameter include the following values:  
  * **USERNAME**: warehouse name is based on username to ensure uniqueness in the namespace  
  * **RANDOM**: warehouse name is based on random words to ensure uniqueness in the namespace  
  * **NOTE:** Any other string will be interpreted as a literal. In this scenario, a shared warehouse will be created and accessible by all users.  
* **secondary** (optional: boolean)  
  If specified, the system will create a secondary principal for use by the user (ie for demonstrating and validating access control scenarios for workspace resources or data governance). Here’s what will happen when this feature is enabled:  
  * a group will be created using the user’s alias (typically, their username with domain dropped. This makes the entity easier to manage from the UI and SQL  
  * a service principal is created within the group  
  * a token to access the workspace through the service principal is created in a secret scope. The scope is named after the primary username, and the secret itself is named “token”

  How to use it:

  * user grants access of something to the group (the group is easier to identify, particularly in SQL statements)  
  * connect to Databricks (ie through a SQL warehouse, SDK, or SQL connector) using the token

* **lakebase** (optional: object)  
  If specified, the system will automatically create and manage a Lakebase instance for each user, applying the parameters specified in this block. For a description of the parameters, refer to the [Databricks REST API for Lakebase instance creation](https://docs.databricks.com/api/workspace/database/createdatabaseinstance). Additional parameters accepted are outlined below. For parameters not supplied, the system will assume sensible defaults.  
  * **name:** instances will be named according to the setting of the **name** parameter. Options for this parameter include the following values:  
    * **USERNAME**: warehouse name is based on username to ensure uniqueness in the namespace. User will have sole access to the database.  
    * **RANDOM**: warehouse name is based on random words to ensure uniqueness in the namespace. User will have sole access to the database.  
    * Any other string will be interpreted as a literal. In this scenario, a shared instance will be created and accessible by all users.  
  * **schema:** if specified, schema within instance will be created. User will have all privileges within the schema (but will not own the schema). User(s) will NOT have permission to create new schemas.  
    If not specified, no schema will be created. User(s) will have permission to create schemas.  
    Schema name should be specified as one of USERNAME or RANDOM (see above).

### Course content

JSON key: **content** (optional: DBC object)

Specifies lab content. This will be extracted into a user’s workspace folder. Note the following:

* Use **entry** to specify the “main” notebook. If not specified, the system will assume the first notebook in the sorted list. ### Datasets

JSON key: **datasets** (optional: list(object))

Specifies datasets to include. In this context, a “dataset” is a product from marketplace formatted specifically for Databricks Academy courses (although can still be generally used by Marketplace customers).

Each dataset in the list is defined by the following parameters:

* **listing\_id** (optional: str): id string of the dataset to install (id as listed on Databricks Marketplace). You must specify this or the **listing\_name** as desired.  
* **listing\_name** (optional: str): name of the dataset to include (name as listed on Databricks Marketplace). You must specify this or the **listing\_id** as desired.  
* **version** (str): version of the dataset being used. This is required.  
* **name** (str): simplified dataset name to present to the learning environment (catalog, file paths, etc). This is required and must match what your courseware expects to see.

Included datasets will appear in the metastore as follows, with all users having read-only privileges over all assets:

* Within a foreign catalog named *dataset\_\<name\>*.  
* Within a schema named *\<name\>\_\<version\>*.

See also data side loading (alternative approach for adding datasets that are not suitable for publication in Marketplace)

See also marketplace (for installing any product from Marketplace)

### Grading

JSON key: **autograde** (optional: DBC object)

Specifies content to execute to assess student’s work (when they click **Submit**). This is executed as a user with administrative privileges at the workspace and metastore level. Note the following:

* Use **entry** to specify the notebook responsible for managing the grading process. This notebook will be run with the following options:  
  * **host**: specifies the workspace URL to use for issuing REST API calls, if your notebook needs to do that  
  * **token**: specifies a bearer token to use for REST API authentication, if your notebook needs to do that.  
  * **username**: specifies the full user id (email) of the user being assessed.  
  * **table**: specifies the name of the table to record the results. For now, the notebook is expected to create this table with the following schema:  
    * **rubric**: VARCHAR(255)  
      The name of the objective being scored (as defined in the Vocareum part configuration)  
    * **score**: INT  
      The value being assessed. Accepted range for this value is 0 through the maximum score for the rubric as defined in the Vocareum part configuration.  
    * **feedback**: VARCHAR(255)  
      Text feedback should be provided in cases where the score value is less than the maximum value.  
* Though the **folder** option is accepted and honored, it has no visible impact to the users of the system and its use here is not generally recommended.  
* Any supplied patches are applied prior to running this job.

### Marketplace

### Secrets

JSON key: **secrets** (optional: object)

If specified, the system will automatically create the named secret scope(s) with READ access enabled for all users of the workspace. The structure of this object consists of one or more objects. Refer to the [common patterns](#add-secrets) for an example.

### Vector search endpoint configuration

JSON key: **shared\_vector\_search\_endpoints** (optional: list of objects)

If specified, the system will automatically create vector search endpoints for each item in the specified list. 

### Direct file access (deprecated)

The infrastructure continues to support direct file access for DBFS or cloud storage, but developers are strongly encouraged to use UC structures instead (volumes, external locations, etc).

JSON key: **enable\_dbfs** (optional: boolean)  
Allows users to access DBFS from the workspace, programmatically and through the UI (assumed true by default).

JSON key: **enable\_files** (optional: boolean but not recommended)  
Allows users to perform legacy file access from their workloads (via the **ANY FILE** privilege) (assumed false by default).

### Personal Access Tokens

JSON key: **enable\_tokens** (optional: boolean)

Allows all users to generate and use personal access tokens for programmatic access to the workspace (assumed false by default).

### Initial redirect

JSON key: **entry** (optional: string)

Specify redirection path, relative to main workspace. This overrides any redirection that may have been triggered by **content**. Some common uses for this include:

* You want to override the “main” notebook picked up in **content** (for example, set **entry** to “” to force landing on main page)  
  * There’s no content but you want to redirect (for example, set **entry** to “sql/editor” to land on DBSQL query editor)

## Common Patterns

This section outlines some common patterns that can be reused in course configuration files to achieve specific results.

### Install Cluster Libraries

If you need a library preinstalled on your learners’ clusters, use the following pattern, taken from *Deep Learning with Databricks*. This installs a required Maven library on the learner’s cluster. The cluster will not accept commands until it’s running and the library has been installed. Note, since it’s defined as a list, this mechanism can accommodate multiple libraries if needed.

  "cluster\_config": {  
    "node\_type\_id": "m4.xlarge",  
    "spark\_version": "12.2.x-cpu-ml-scala2.12",  
    **"libraries": \[**  
      **{**  
        **"maven": {**  
          **"coordinates": "com.johnsnowlabs.nlp:spark-nlp\_2.12:4.2.5"**  
        **}**  
      **}**  
    **\]**  
  }

### Specify DBR

By default, the framework will choose the most recent standard LTS DBR. If you need a specific version, use **spark\_version** as shown below. This example is taken from *Deep Learning with Databricks*. 

  "cluster\_config": {  
    "node\_type\_id": "m4.xlarge",  
    **"spark\_version": "12.2.x-cpu-ml-scala2.12"**  
  }

### Specify Node Type

By default, the framework will choose the most conservative node type it can. If your course has specific memory/CPU requirements that are not met by this choice, then you can override this setting using **node\_type\_id** as shown below. This example is taken from *Deep Learning with Databricks*. 

  "cluster\_config": {  
    **"node\_type\_id": "m4.xlarge",**  
    "spark\_version": "12.2.x-cpu-ml-scala2.12"  
  }

### Specify Spark Configuration Settings

If you need to set Spark options, use the pattern as shown below. 

  "cluster\_config": {  
    "spark\_version": "14.0.x-scala2.12",  
    **"spark\_conf": {**  
      **"spark.driver.maxResultSize": "0"**  
    **},**  
    "node\_type\_id": "i3.4xlarge"  
  }

### Replace Notebook/File in DBC

Sometimes, you might need to make a minor adjustment to a resource within your course material for it to work properly in the Vocareum environment. To this, use the patching mechanism as illustrated in these examples. Here we highlight two common use cases.

#### Patching workspace setup

Sometimes, minor differences in the Vocareum environment invalidate one or more commands in the *Workspace-Setup* notebook. If this is the case, you can edit the notebook (commenting out or changing the failing commands) and patch the DBC by overlaying the edited version onto the version that gets extracted from the DBC. Here, in this example from *Data Engineering with Databricks*, we show how to do this. The following example replaces the notebook *Includes/Workspace-Setup* with the *Workspace-Setup.py* file that we upload to the *patches* subdirectory of the course folder. Note, since it’s defined as a list, this mechanism can accommodate multiple patches if needed.

  "workspace\_setup": {  
    "src": "data-engineering-with-databricks.dbc",  
    "entry": "Includes/Workspace-Setup",  
    **"patch": \[**  
      **{**  
        **"src": "patches/Workspace-Setup.py",**  
        **"target": "Includes/Workspace-Setup"**  
      **}**  
    \]  
  }

#### Patching user content

Sometimes, we need to make hotfixes to course content to adjust for the Vocareum environment or simply to fix critical errors that have been identified by learners or instructors. Generally this should be regarded as a temporary measure until the upstream source can be fixed and a new DBC can be published that fixes the error, but hotfixing in this manner provides a good short-term solution. Here, in this example from *Introduction to Python for Data Science and Data Engineering*, we replace the *ITP 00 \- Databricks Environment* notebook (the one found in the regular and solution folders) with a patched version using a list.

  "content": {  
    "src": "introduction-to-python-for-...-and-data-engineering.dbc",  
    "entry": "AGENDA",  
    **"patch": \[**  
      **{**  
        **"src": "patches/ITP 00 \- Databricks Environment.py",**  
        **"target": "Solutions/ITP 00 \- Databricks Environment"**  
      **},**  
      **{**  
        **"src": "patches/ITP 00 \- Databricks Environment.py",**  
        **"target": "ITP 00 \- Databricks Environment"**  
      **}**  
    **\]**      
  }

If you need to add secrets for all to read, use the pattern as shown below.   
    
  "secrets": {  
    "secret-scope-name": {  
      "key1": "secret1 string",

### Create Vector Search Endpoints

Use the following pattern to add up to ten fixed-name vector search endpoints.

  "shared\_vector\_search\_endpoints": \[  
      {  
          "name": "dbacademy-endpoint1",  
          "endpoint\_type": "STANDARD"  
      },  
      {  
          "name": "dbacademy-endpoint2",  
          "endpoint\_type": "STANDARD"  
      }  
  \]

### Deploy a model

Use the following pattern to deploy a model to a serving endpoint which all users will have permission to query. Because it’s an array you can deploy multiple models if needed.

  "models": \[  
      {  
          "name": "\<desired name of endpoint\>",  
          "model": "\<catalog\>.\<schema\>.\<model\>"  
      }  
  \]

### Enable a System Schema

Use the following pattern to enable a system schema (in this case the *access* schema to access the audit logs).

  "metastore\_config": {  
    "system\_schemas": \[  
      "access"  
    \]  
  }

### Marketplace Access Defaults

Marketplace access is now disabled by default. All account users have USE\_MARKETPLACE\_ASSETS revoked unless explicitly enabled.

To enable, add in your config:

"metastore\_config": {  
	"enable\_use\_marketplace\_assets": true  
}

If this block is omitted or set to false, users cannot pull any datasets from the Marketplace.

### Workspace Setup – Serverless Job Cluster

Workspace setup now supports running jobs on a serverless job cluster.The config key "serverless\_job\_cluster" is read from "workspace\_setup" in config.json.

If set to true, the job runs on a serverless job cluster. If false or omitted, a new cluster is created via the new\_cluster parameter.
