Chapter 4

One Command.
Everything Configured.

No clicking through UIs. No copying resource IDs. A single script wires up every Databricks resource.

1

Install dependencies

Pull the entire Python environment.

uv sync
2

Run the setup wizard

Validates credentials, configures Databricks resource IDs → saves everything to .env.local.

./run setup
3

Create the data layer

Auto-creates UC schema, Delta tables, SQL functions & stored procedures.

uv run python data/init/create_all_assets.py
4

Start everything locally

Launches backend (8000), API (3001) and React frontend (3000) in one shot.

bash scripts/start_local.sh

Smart .env.local

  • Reads, verifies, and writes every config value
  • Existing resources detected and reused — no duplicates, no overwrites

Streaming by Default

  • Streams agent responses token by token
  • See the live UX exactly as it behaves in production
Chapter 5

in One Command

The deploy script handles everything: env sync, bundle validate, deploy, app startup, and permission grants.

1

Validate first (optional)

Checks all env vars, bundle config, and Python imports without touching Databricks.

Deploy

Syncs config, deploys the bundle, starts the app, and grants UC + warehouse + secret access — automatically.

Open the app

Chapter 7

Add Your
Own Data

Replace the reference dataset or extend it. Two files are all it takes — no script changes needed.

1

Drop a CSV

Add your seed data to data/csv/<table_name>.csv — up to 100 rows of realistic synthetic data.

2

Write the DDL

Create data/init/create_<table_name>.sql with your column types. Use __SCHEMA_QUALIFIED__ as the table prefix — resolved at runtime.

3

Provision it

Run the SQL file directly, or let create_all_assets.py auto-discover and create it.

uv run python data/py/run_sql.py data/init/create_<table>.sql

Auto-Discovery

  • create_all_assets.py scans data/csv/ and runs the matching SQL for every CSV it finds
  • No orchestrator edits required

Schema-Agnostic SQL

  • __SCHEMA_QUALIFIED__ substituted at runtime from your environment
  • The same SQL works in any workspace

Genie Picks It Up

New tables are included in the Genie space on its next run — the agent can query them in natural language immediately.

Chapter 8

Add Your Own Tools

One file per tool. One import. One list entry in agent/agent.py. Three patterns cover every case.

Pattern 1

SQL Read

  • Write a SQL template in data/func/
  • Call it from a @tool function via the warehouse
  • Precise, fast, auditable structured queries
Pattern 2

Action (Stored Procedure)

  • Call a UC stored procedure in data/proc/
  • Use when the agent needs to write back — update status, trigger workflows, mutate state
Pattern 3

Knowledge Assistant

  • HTTP POST to a KA serving endpoint
  • Agent routes here for unstructured documents — regulation text, manuals, FAQs
Chapter 9

Knowledge
Assistants

Back the agent with your own documents. A YAML config and a folder of PDFs is all you need.

1

Write the YAML config

Define display name, instructions, and knowledge sources in config/ka/ka_<slug>.yml. PDFs must be in your UC Volume ({volume_path} placeholder).

2

Validate and deploy

Dry-run first, then deploy and wait for ACTIVE. The endpoint is written to .env.local automatically.

uv run python scripts/ka/create_kas_from_yml.py config/ka/ka_<slug>.yml
3

Wire as an agent tool

Add a KA tool in tools/ using the HTTP pattern, register it in agent/agent.py. The agent now knows when to route to it.

YAML-Driven

  • Display name, instructions, output format, knowledge sources — all in one config file
  • Shared output_format.yml merged automatically

Endpoint Auto-Wired

When the KA reaches ACTIVE state, its serving endpoint is written to .env.local as PROJECT_KA_<SLUG> — ready to use in the tool with no manual copy-paste.

Agent Routes Automatically

  • Agent reads each tool's docstring to decide when to invoke it
  • Write a precise docstring — no extra routing logic needed
Chapter 10

Evaluate
& Improve

Measure answer quality objectively before shipping a change. A custom LLM judge scores every response.

1

Run the baseline

Score the KA against a curated test dataset. Results land in MLflow.

uv run python eval/run_eval.py
2

Change something

Update the KA instructions or add a citation guideline. No redeploy needed — the KA picks up instruction changes live.

3

Run the comparison

Score again and compare both runs side-by-side in the MLflow UI.

uv run python eval/run_eval.py --run2

Custom LLM Judge

  • Scorer cites_regulation_precisely uses Claude to check each answer
  • 1.0 = PASS, 0.0 = FAIL — no ambiguity

MLflow Tracking

  • Every eval run logged as an MLflow experiment run
  • Compare scores, inspect predictions, track improvement over time

Reset & Iterate

  • Run ./run reset-workspace to start fresh without losing catalog or KAs
  • Tables, procedures, Genie space rebuilt on next setup
